# 项目设计文档 · lan-boardgame

> 版本 2.0 | 最后更新 2026-06-19

---

## 1. 项目概述

局域网联机对战棋类平台，支持多人观战、聊天、AI 对弈、棋谱研究。以围棋为首发游戏，架构设计面向多游戏扩展。

**项目名**：`lan-boardgame`

---

## 2. 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端框架 | React 18 + TypeScript | 组件化 UI 管理 |
| 构建工具 | Vite | 开发服务器 + 打包 |
| 后端 | Node.js + TypeScript | 事件驱动，适合 WebSocket |
| 数据库 | SQLite (via sql.js) | 零配置，单文件部署 |
| 密码加密 | bcrypt 10 轮盐值 | 安全存储用户凭证 |
| 实时通信 | WebSocket (ws 库) | 房间同步、聊天、对局 |
| AI 增强 | KataGo GTP 引擎 | 围棋 AI 增强（19×19/13×13/9×9） |

### 3.1 游戏列表

| 游戏 | 状态 | 说明 |

| 🟢 五子棋 (Gomoku) | ✅ 已开发完成 | 内置 AI（3级难度） |
| 🟢 中国象棋 (Chinese Chess) | ✅ 已开发完成 | 内置 AI（3级难度，Minimax α-β 剪枝） |
| 🟢 国际象棋 (Chess) | ✅ 已开发完成 | 内置 AI（3级难度，Minimax α-β 剪枝） |
| 🟢 国际跳棋 (Draughts) | ✅ 已开发完成 | 内置 AI（3级难度） |

大厅展示所有游戏卡片，全部游戏均可正常游玩。

### 3.2 围棋规则

- **中国规则 (数子制)**：默认规则，终局数盘上活子数，贴 3¾ 子（7.5 目）
- **日韩规则 (数目制)**：可选规则，终局数目 + 死子计分，贴 6.5 目
- 两种规则引擎分开存放，不耦合
- 创建房间时可选规则，默认中国规则

### 3.3 房间系统

房间系统有**两层状态**：房间自身的生命周期状态 + 房间内的活动状态。

#### 3.3.1 房间生命周期状态（Room Lifecycle）

房间只有两种生命周期状态：

```
  创建房间
      │
      ▼
  ┌──────────┐    房主离开 → 全员弹窗「👑 xxx 离开房间」
  │  ACTIVE  │ ──────────────────────────────────────────► DESTROYED
  └──────────┘                房间立即销毁，所有人回到大厅
```

- **ACTIVE**：房间活着。只要房主不离开，房间一直存在
- **DESTROYED**：房主已离开，房间已销毁。所有人收到弹窗后回到大厅

#### 3.3.2 房间活动状态（Room Activity）—— 空闲的 3 种变体

房主在当前房间内的活动状态，根据数据统计的需要精确区分：

```
  ┌───────────────────────────────────────────────────────┐
  │                    空闲（Idle）                        │
  │                                                       │
  │  ├─ 0: 完全空闲（刚进入房间，什么都没干，可聊天）       │
  │  ├─ 1: 打谱中（载入棋谱研究）                           │
  │  └─ 2: AI 对弈中（和 AI 下棋）                         │
  │                                                       │
  │  └─ 等人中：房主什么都没干，等待其他人加入                │
  │     （统计归类为 空闲 0，因为行为 == 完全空闲）           │
  └────────────────────┬──────────────────────────────────┘
                       │
        申请人进入 → 「申请对局」
                       │
              ┌────────┴────────┐
              │                 │
              ▼                 ▼
        房主同意         房主拒绝/60秒超时
              │                 │
              ▼                 ▼
        PVP 对局中         回到原来的空闲状态
              │           （0/1/2 不变）
              │
        对局结束
              │
              ▼
         回到空闲 0（房主重新选）
```

#### 3.3.3 房间布局

```
┌─────────────────────────────────────────────┐
│  房名: xxx 的房间       游戏: 围棋   规则: 中国 │
├──────────┬──────────────────┬────────────────┤
│ 人员列表  │                  │                │
│          │     棋盘         │   聊天区       │
│ 👑 房主   │                  │                │
│  🧑 玩家2│   (Canvas)       │   消息列表     │
│  👤 观战3 │                  │   输入框       │
│  👤 观战4 │                  │                │
├──────────┴──────────────────┴────────────────┤
│  底部信息栏：状态/时间/操作按钮               │
└─────────────────────────────────────────────┘
```

#### 3.3.4 房间核心规则

- 房主创建房间，**房间依附房主存在**
- 房间人数上限：对局 2 人 + 不限观战（最多显示 50 人）
- 房间满 2 人后仍可继续进入（进入观战模式）
- 房主在人员列表最上面，显示 👑 前缀

#### 3.3.5 房主离开

**房主离开** → 全员弹窗「👑 xxx 离开房间，房间即将销毁」→ **房间立即销毁**，所有人回到大厅

#### 3.3.6 玩家离开

**非对局中的普通玩家/观战者离开** → 仅刷新左侧人员列表，不弹窗

#### 3.3.7 对局中对手离开

**对局中的一方离开** → 全员弹窗「xxx 离开房间，电脑即将接管」→ 🤖 AI 替换棋盘上该玩家的名字位置，AI 接管继续对局

#### 3.3.8 单人模式与加入者可见

房主创建房间后可选的状态（即三种空闲）：

| 房主状态 | 加入者刚到房间时 | 加入者可操作 |
|---------|----------------|------------|
| 空闲 0（等人中） | 空棋盘 + 聊天 | 坐下对局 + 聊天 |
| 空闲 1（打谱中） | 看到棋盘上棋谱 + 聊天 | **观战 + 「申请对局」+ 聊天**，不能操作棋盘 |
| 空闲 2（AI 对弈中） | 看到棋盘上 AI 对局 + 聊天 | **观战 + 「申请对局」+ 聊天**，不能操作棋盘 |

**重要**：空闲 1 和 2 时，加入者只能看，不能碰棋盘。只有通过「申请对局」流程才能开始 PvP。

#### 3.3.9 「申请对局」详细流程

**触发条件**：
- 只有**房主处于空闲状态**（0/1/2 均可，包括等人中）时，其他人才可「申请对局」
- 房主 PvP 对局中，**不能申请**，只能聊天

**流程**：

```
申请人点击「申请对局」
         │
         ▼
  ┌──────────────────────────────────┐
  │          申请人收到弹窗           │
  │                                  │
  │  已经向 👑xxx 申请对局             │
  │                                  │
  │  ┌──────────────────────────┐    │
  │  │  ⏳ 60秒倒计时            │    │
  │  └──────────────────────────┘    │
  └──────────────────────────────────┘
  ┌──────────────────────────────────┐
  │          房主收到弹窗             │
  │                                  │
  │  xxx 申请与您对局                 │
  │                                  │
  │  ┌──────┐    ┌──────┐          │
  │  │ 同意  │    │ 拒绝  │          │
  │  └──────┘    └──────┘          │
  └──────────────────────────────────┘
         │                    │
         │ 房主点「同意」      │ 房主点「拒绝」或 60 秒超时
         ▼                    ▼
  ┌─────────────────┐   ┌─────────────────────────┐
  │ 立即强制刷新棋盘  │   │ 房主弹窗消失            │
  │ 申请人拉到对面    │   │ 继续原来的空闲状态       │
  │ PVP 对局开始     │   │ （0/1/2 不变）           │
  └─────────────────┘   │                         │
                         │ 申请人弹窗变化:           │
                         │ 「👑xxx 拒绝对局」        │
                         │ 申请人回到观战+聊天状态   │
                         └─────────────────────────┘
```

**关键约束**：
- 弹窗**只有申请人**和**房主**两人能看到，房间内其他人看不到
- 60 秒倒计时期间双方弹窗都在，超时后**双方弹窗同时消失**
- 房主拒绝后，**房主的空闲状态不变**（打谱的继续打谱，AI 的继续 AI）
- 房主同意后，**当前空闲活动立即终止**，棋盘直接被 PvP 对局覆盖

#### 3.3.10 PvP 终局后的「再战一局 / 退出」流程

PvP 对局结束后，弹窗给**对局双方**（观战者看不到这个弹窗）：

```
         PvP 对局结束
              │
              ▼
    ┌─────────────────────┐
    │   弹窗（对局双方）    │
    │                     │
    │  ┌──────┐ ┌──────┐ │
    │  │再战一局│ │ 退出  │ │
    │  └──────┘ └──────┘ │
    │     ⏳ 60 秒倒计时   │
    └─────────┬───────────┘
              │
       ┌──────┴──────┐
       │             │
       ▼             ▼
    ┌──────┐    ┌──────────┐
    │A点再战│    │A点退出   │
    │      │    │或60秒超时 │
    └──┬───┘    └────┬─────┘
       │             │
       ▼             ▼
  A的按钮变禁用:    A退出对局状态
  「已申请」       回到房间状态
       │
       ▼            房主退出: 棋盘保留终局画面
  B的按钮变为:     申请人退出: 回到观战状态
  「对方申请再战」
       │
       ▼
  B点击「对方申请再战」
       │
       ▼
  两人都同意
       │
       ▼
   新一局立即开始！
```

**规则明细**：

| 条件 | 结果 |
|------|------|
| 一方点「再战一局」 | 己方按钮变灰「已申请」，对方按钮变为「对方申请再战」 |
| 对方点「对方申请再战」 | 双方同意 → **新一局立即开始** |
| 一方点「退出」 | 该方退出对局状态，回到房间 |
| 房主点「退出」 | 回到房间，**棋盘保留终局画面** |
| 申请人（非房主）点「退出」 | 回到房间观战状态 |
| 对方已点「退出」时，剩余一方还在弹窗中 | 「再战一局」→ 变为「对方退出」（灰色不可点击），短暂停留后自动关闭，剩余一方回到房间 |
| 双方 60 秒无操作 | 自动退出，等同于双方各自点了「退出」 |

**一方退出时的同步处理**：

当一方点了「退出」后，另一方的弹窗同步变化：

```
对方点了「退出」
       │
       ▼
原来「再战一局」按钮 → 变为「对方退出」（不可点击，灰色）
弹窗不再有倒计时动作
短暂停留后自动关闭
       │
       ▼
剩下一方回到房间状态
```

#### 3.3.11 用户在房间中的状态

用户在房间内有两个维度：

**维度一：身份（在房间中固定不变）**

| 身份 | 图标 | 说明 |
|------|------|------|
| 房主 (Owner) | 👑 | 创建房间的人，他离开→房间销毁 |
| 玩家 (Player) | 🧑 | 坐在棋盘前对弈的人（0~2 人） |
| 观战 (Spectator) | 👤 | 进入房间看棋聊天，不下棋 |

**维度二：活跃状态（动态变化）**

| 状态 | 触发条件 | 表现 |
|------|---------|------|
| 在线 (connected) | 正常 WebSocket 连接 | 正常下棋/观战/聊天 |
| 断线 (disconnected) | 网络断开 | 等待 90 秒（仅对局中的玩家受影响） |
| AI 接管 (ai_controlled) | 对局方断线超时 或 主动离开 | 棋子旁边显示 🤖 代替原玩家名 |
| 申请对局中 (challenging) | 申请人已发起申请，等待房主回应 | 显示 60 秒倒计时弹窗，不可再次申请 |
| 被申请人挑战中 (being_challenged) | 房主收到申请 | 显示同意/拒绝弹窗 |

**身份 × 活跃状态组合示例**：

| 身份 | 在线 | 断线 | AI接管 | 申请中 |
|------|------|------|--------|--------|
| 房主 | ✅ 正常 | ✅ 弹窗→房间销毁 | 不适用 | ✅ 收到申请弹窗 |
| 玩家 | ✅ 正常下棋 | ✅ 等90秒→AI接管 | 🤖 显示 | ✅ 可申请（空闲时） |
| 观战 | ✅ 聊天 | ✅ 仅自己断开 | 不适用 | ✅ 可申请（空闲时） |

### 3.4 棋盘视角

对局时，每位玩家看到的棋盘**自动反转**，确保自己的棋子在屏幕下方，与现实中围棋对局视角一致。观战者看到的是**黑棋在下方**的默认视角。

### 3.5 聊天系统

- **范围**：仅在房间内可见
- **参与者**：房主、对局双方、观战者均可发言
- **可见性**：后进入房间的人，只能看到自己进入时间点之后的消息，看不到历史消息
- **无全局大厅聊天**

### 3.6 打谱（棋谱研究）

两种模式，两者都支持：
1. **载入历史对局复盘**：加载数据库中的历史对局记录，一步步浏览
2. **载入标准棋谱研究**：加载 SGF 等标准格式棋谱，支持手动摆变化图

### 3.7 AI 系统

- **水平**：中级（能进行基本的形势判断，非随机走棋）
- **策略**：每个游戏的 AI 独立开发，不共用 AI 算法
- **触发**：AI 对弈（单人模式）/ 对手断线 AI 接管

#### KataGo 集成（仅围棋）

围棋支持 KataGo GTP 引擎增强 AI：

- **架构**：服务端通过 GTP 协议启动 `katago.exe` 子进程，每个房间独立会话
- **配置**：支持 19×19 / 13×13 / 9×9 棋盘，中国规则 / 日韩规则可选
- **性能**：基于 28-block 神经网络权重（~271MB），OpenCL 推理
- **降级**：KataGo 出错时自动 Pass，不中断对局
- **生命周期**：对局结束时自动销毁 KataGo 子进程

## 4. 用户系统

### 4.1 角色

| 角色 | 登录入口 | 权限 |
|------|---------|------|
| 普通用户 | `/` | 游戏大厅、房间、对局、聊天 |
| 管理员 | `/admin` | 后台管理面板 |

### 4.2 登录

- **两个独立登录页面**，不复用组件
- 普通用户：`/login` → 游戏大厅 `/`
- 管理员：`/admin/login` → 后台 `/admin`
- 注册仅普通用户，管理员账号预置或通过数据库直接创建
- 密码使用 bcrypt 10 轮加密

### 4.3 管理后台

| 模块 | 功能 |
|------|------|
| 总览 (Dashboard) | 在线用户数、活跃房间数、今日对局数等概览统计 |
| 用户管理 | 用户列表、封禁/解封、角色修改 |
| 对局记录管理 | 历史对局列表、查看棋谱回放、删除记录 |
| 游戏管理 | 各游戏开关、规则默认设置 |
| 数据统计 | 对局量趋势、游戏热度、用户活跃度图表、房主空闲类型分布（0/1/2 占比） |

## 7. 数据流：典型场景

### 场景 A：普通用户对局（完整路径）

```
1. 用户 A 访问 /login → 登录 → 跳转大厅 /
2. 点击「围棋」卡片 → 弹出小窗
3. 点击「创建房间」→ 房间创建成功 → 进入房间
4. 房间活动状态变为「空闲 0」，可选的三个行为：
   ├─ 什么都不做，等人加入 → 保持空闲 0
   ├─ 点「打谱」→ 状态变为空闲 1
   └─ 点「AI 对弈」→ 状态变为空闲 2
5. 用户 B 登录 → 大厅 → 点击「加入房间」→ 进入房间

   情况 A：房主空闲 0（等人中）
      用户 B 直接坐下 → PVP 对局开始
      
   情况 B：房主空闲 1（打谱中）或空闲 2（AI 对弈中）
      用户 B 只能观战 + 聊天
      用户 B 点击「申请对局」→ 给房主发申请
      房主同意 → 强制切换为 PVP 对局
      房主拒绝/超时 → 各自继续当前状态

6. PVP 对局开始 → WebSocket 双向同步 → 棋盘自动反转
7. 对局中一方断线 → 弹窗「xxx 离开房间，电脑即将接管」→ AI 接管
8. 终局 → 显示结果 → 保存对局记录到数据库
9. 回到空闲 0（房主可重新选）
```

### 其他说明

#### AI 难度等级与人类棋手水平对照

| 游戏 | 简单 | 普通 | 困难 |
|------|------|------|------|
| **围棋** | ~10-15 级（基础评分：提子+领地+位置） | ~业余 1-2 段（完整启发式：提子/逃子/打吃/领地/压制/位置/开局/填眼/劫） | ~业余 3-5 段（MCTS 蒙特卡洛树搜索，800 次迭代 + UCB1 + 领地估算 rollout） |
| **五子棋** | ~入门级（仅进攻评分，不考虑防守和位置） | ~中级（进攻+防守+中心加成） | ~高级（进攻+强防守×1.3+中心加成×1.5） |
| **中国象棋** | Minimax 深度 1（约入门级） | Minimax 深度 2（约初级水平） | Minimax 深度 3-4（约中级水平） |
| **国际象棋** | Minimax 深度 1（约入门级） | Minimax 深度 2（约初级水平） | Minimax 深度 3-4（约中级水平） |
| **国际跳棋** | 贪心搜索（仅吃子+位置评分） | 深度受限搜索（2 层 + 静态评估） | 深度受限搜索（4 层 + 动态评估） |

**围棋 AI 算法说明**：
- **简单/普通**：基于启发式评分函数，对每个空位计算提子、逃子、打吃、领地、压制、位置价值等分数，选择最高分位置
- **困难**：MCTS（蒙特卡洛树搜索），通过模拟对局估算胜率，使用 UCB1 公式平衡探索与利用，800 次迭代后选择访问次数最多的走法

**五子棋 AI 算法说明**：
- 基于连线评分：计算每个空位落子后形成的连子数（2连、3连、4连、5连），同时评估进攻和防守价值
- 难度通过调整搜索范围（邻近距离）和评分权重（防守系数、位置加成）来区分

**中国象棋 AI 算法说明**：
- Minimax 搜索 + Alpha-Beta 剪枝
- 评估函数：棋子价值（车=900, 炮=450, 马=400, 仕/相=200, 兵=100, 帅=10000）+ 位置价值表
- 难度通过搜索深度区分（1-4 层）

**国际象棋 AI 算法说明**：
- Minimax 搜索 + Alpha-Beta 剪枝
- 评估函数：棋子价值（后=900, 车=500, 马=320, 象=330, 兵=100, 王=20000）+ 位置价值表 + 机动性评估
- 难度通过搜索深度区分（1-4 层）

**国际跳棋 AI 算法说明**：
- 基于深度受限的搜索树 + 静态局面评估
- 评估函数：子力优势（普通棋子=100, 王棋=300）+ 位置价值 + 中心控制 + 升王潜力
- 简单模式：贪心搜索（仅当前最佳吃子/走法）
- 普通模式：2 层深度搜索 + 静态评估
- 困难模式：4 层深度搜索 + 动态评估
